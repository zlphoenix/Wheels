# 功能要点整理

## 门户网站
### 整体描述
网站作为企业对用户的门户，起到介绍公司情况，发布产品动态，最新优惠活动的主要平台，以此吸引用户来购买产品和服务。

因为公司定位于高端汽车轮毂产品销售和改装服务，并提供汽车美容，保养等配套服务，所以网站需要专门对这些用户进行优化，主要是通过预览轮毂效果，并立即下单预约安装，来提升销售量。

网站要与其他推广工具如微信，微博，淘宝建立紧密的联系，互相之间建立连接，可以互相拉动流量

### 参与者介绍：
1. 网站管理员：华丽公司员工，日常负责维护网站数据，提交新产品信息，及时了解轮毂预定情况并为用户上门安装做准备。并在销售完成后，将预约工单结束。
2. 用户：所有浏览主页的客户都属于网站用户范畴，所有用户都可以浏览网站内容，查询商品信息。
	1. 其中，未注册用户为游客，只能执行浏览操作，无法通过网站进行互动。
	2. 通过身份认证的游客成为注册用户，可以通过网站上传个人汽车图片，并搭配产品进行预览，满意后可以直接在网站上下单预约安装。后期加入社交网络分享功能后，可以在安装前分享预览效果图，也可以在安装后分享实际效果图。并提供打分评价，内容评论功能，帮助提高影响力。
	3. 需要讨论`通过一定的身份认证机制，比如参加推广活动，累计消费超过一定额度等，注册用户可以提升为VIP会员，可以通过星级评定来区分不同层次的vip用户。此类用户可以享受一定的价格和服务优惠`
3. 与网站有数据和服务交互的系统
	1. 基地系统：提供商品基础数据，库存余额
	2. 社交网络系统如微博，微信：建立绑定账号，提供分享和推广功能
	3. 在线支付系统：如支付宝，财付通，用于在线支付的支持
	
===
## 详细用例清单
### 用例UC01 发布网站内容

**参与者，及其关注点**

1. 网站管理员：能够及时，方便地录入已有的轮毂商品信息
2. 基地系统：提供商品基本信息和库存余额数量

**前置条件**

1. 基地系统中的存储模型，数据字典需要的到确认，含义必须明确
2. 管理员通过管理后台登录网站系统
3. 不能受到仓库停电的影响，需要保持对基地系统的连接

**完成依据**

数据内容被正确发布到网站上

**主要场景**

1. 管理员登录网站管理后台
2. 从现有的基地系统，或者已经发布的商品信息中查询出需要发布的产品
	1. 可以将已经不再销售的商品从清单中排除，以后普通查询将不再显示此记录
	2. 如果需要查看已经被废弃的商品可以指定显示所有结果
	3. 对于已发布的商品，检查库存余额，如果库存余额小于安全库存量，提示是否下架或预订处理的提示
	4. 对于已经存在预约订单，但是库存量小于预约订单数量的提示告警信息，提前处理可能影响履约的情况，可以选择从供应商采购，或者参考订单延期处理[UC05](#UC05)
3. 维护商品需要展示给用户看属性，如商品名称，品牌，分类，适用车型，尺寸，规格，价格等 
4. 上传轮毂预览图（用于在线预览安装效果），展示图（多种分辨率和角度，用于用户查询型号时查看外观）；
	1. 可以根据规则批量上传，比如商品名称与图片文件名一致，展示图与预览图通过后缀来区分
5. 编写商品介绍文章
6. 发布数据：可以选择商品生效时间，在此时间之后用户可以在网站和移动客户端看到 重复3～6，直到本次发布结束
7. 对于已经不再销售的产品或者临时无法预约的产品，需要执行下架处理，可以选择立即下架或者在指定时间下架。下架后的商品不能再被预约和订购。
8. 调价：可以对已发布的商品做调价处理，调价可以立即生效也可以指定在未来某个时间生效，可以用于促销活动

**其他需求**

尽量减少用户输入，现有系统中有的信息尽量抽取出来，而不是每个都要手工输入

**发生频率**

系统上线前后会比较频繁发生，一旦基础数据导入结束，进入日常工作状态，可能会根据新产品的推出速度来决定其发生频率

**风险识别**

* 对现有系统中数据的读取，需要了解基地系统的数据模型
* 内网中的数据发布到外网（推），还是从外网建立穿透机制访问内网（拉），或者保持长连接
* 内网停电带来的影响导致有些数据不能及时反馈回到内网系统中

### 用例UC02 浏览网站内容
**参与者，及其关注点**

1. 用户：了解企业的情况以及产品情况，能够通过地址访问到网站的人(或者机器人程序)都可以作为网站的用户，他们可以无差别地获取到网站上已经发布地所有内容数据，但是不能做出任何修改动作只能浏览

**前置条件**

1. 网站各板块已经设置完成，而且内容已经发布
2. 通过地址访问到网站，或者从其他途径导航到网站(如搜索，微信，淘宝导航等)

**完成依据**

打开站点页面

**主要场景**

1. 首页展示公司信息，如Slogan，近期优惠和主推的几款明星产品，公司的资质，设备情况总体介绍
2. 浏览轮毂产品，并能根据汽车品牌，型号，轮毂品牌，型号，规格等信息查询。界面可以参考淘宝等电商的查询页面
	1. 查询结果以列表或者表格方式展现，每条产品信息提供一张大图和品牌型号简介
	2. 点击图片或者简介，进入产品细节，其中包含详细的规格参数，最好包含安装效果图
	3. 未来提供产品评价系统后显示已经评价的情况，如果有晒单和经验分享也可建立相应的连接
3. 详细信息中包含预览按钮，如果当前用户已经登录点击后进入轮毂预览，参考 [用例UC04](#UC04)，否则进入登录或注册界面，参考用 [用例UC03](#UC03)
4. 车体美容保养板块显示一辆整车，鼠标移动到指定位置显示相应的美容保养业务介绍，可以参照Bing主页
	1. 点击关注的保养业务点可以查看相应的细节，包括价格，耗材介绍等信息（需要确认，保养到底做了什么）
5. 选中轮毂产品或者保养服务，可以在详细信息页面直接预定，参考[用例06]

**其他需求**

由于直接面向终端消费者，需要尽量考虑用户的易用性需求，美观，具有视觉冲击力，交互友好，方便查询

**发生频率**

一旦业务展开访问量可能会逐渐提升。

### 用例UC03 注册和登录

为了能够为后续的预约 用例UC05 提供依据，更重要的是统计用户信息，用来做后续的数据分析，所以需要用户填写尽量完整的用户信息。其中注册阶段用于统计最基本的手机信息，注册成功后可以选择填写一些个人基本信息并绑定社交网络账号，用于分享预览图和晒单

**参与者，及其关注点**

1. 游客：尚未注册和登录的用户，因为对产品信息感兴趣，需要注册账户

**前置条件**

1. 用户尚未注册账号

**完成依据**

成功创建账户

**主要场景**

1. 游客点击主页上的登录按钮，或者通过轮毂预览进入受限访问内容时，进入注册功能
2. 首先需要用户输入手机号码点击获取验证码
3. 系统通过短信服务商发送验证短信
3. 用户手机上收到短信后填写在注册页面上
4. 如果与发送的一致，账号被建立
5. 进入用户基本信息维护页面
	1. 默认用户账号是手机号码
	2. 可以选择填写网站个人账号，个人社交网络账号，并导入个人基本信息（此处需要与新浪微博等SNS API通讯，获取用户认证授权）此处是可选的
	3. 用户可以填写一些其他的个人信息，如生日（提供生日当天的优惠，鼓励填写，用于统计年龄，数据分析用）车型，Email地址等（`这个需要确认，到底客户的那些数据是对消费有关键的作用的`）
	4. 登录成功后可以向用户邮箱中发送带有公司微博公众账号二维码的慰问邮件(促销动作)
6. 注册完成后，可以适用手机号，

**其他需求**

**发生频率**
**风险识别**
* SNS Api 接口的约束
* 业务：哪些个人信息对未来推广至关重要，需要摸索。用户如果提交的个人信息不够充分，不足以分析数据

### 用例UC04 轮毂预览

**参与者，及其关注点**

注册用户：希望看到自己的车搭配已选择则轮毂效果如何

**前置条件**

1. 用户如果通过导航或者链接来到预览页面，此时应该可以带过来之前选择好的轮毂型号
1. 用户自行准备车体照片
2. 客户端有上传文件的能力，如果是移动设备，访问站点可能会受到限制

**完成依据**

用户能够看到自己汽车的图片上合适的位置摆放自己选择的轮毂

**主要场景**

1. 从微信或者其他推广方式导航来的场景，应该携带了已经选择的轮毂
2. 如果没有，可以根据过滤条件选择自己喜欢的轮毂
3. 上传本地图片，或者从一个已有的链接打开图片，对于比较流行的改装车型号，系统可以内置一些汽车照片
4. 选中或者上传的可以放在个人相册中，以备后续预览使用
5. 以选中的汽车照片为背景，叠加上轮毂正视图照片
6. 可以通过手势移动轮毂位置
	1. 可以锁定两轮毂一起调整或者独立调整每个轮毂
	2. 锁定模式下可以分别调整 轮毂直径，间距，圆心在画布上的坐标
7. 调整好的预览图可以保存下来（不需要保存轮毂图片）
	1. 记录轮毂型号，每个轮毂的圆心坐标，直径就能还原当前预览图
	2. 可以选2～3款预览图进行对比
8. 如果之前绑定了微博账号可以将预览图分享到自己的时间线上，此时需要生成一个预览图
9. 如果对选择的轮毂比较满意，在预览页面直接下单预约 参考 [用例05 预约](#UC05)

**其他需求**

### 用例UC05 预约
**参与者，及其关注点**

1. 注册用户：选中的产品和服务后通过预约生成预约订单，安排日期和时间到零售店进行安装和保养服务（`是否采用购物车机制，需要讨论`）

**前置条件**

1. 用户成功注册为注册用户
2. 选择了至少一件产品和服务(可能通过购物车来暂存)
3. 进入预约页面

**完成依据**

预约成功，生成预约订单，订单处于已审核状态

**主要场景**

1. 确认商品或者服务的信息，如商品型号，数量；服务种类或者套餐类型
2. 确认用户信息，此时如果之前用户提供了姓名和联系电话，Email，SNS账号，车型，车牌号，可以将这些信息从用户档案信息中带过来，如果没有，则需要输入关键的信息：姓名，电话，车型，车牌号`需要找客户确认，这些信息是否充要`
3. 确认到店时间，前期用户数量比较少，可以使如下规则安排
> 目的：能够保证服务可以被满足
> 
> 1. 必须预约一个未来的时间，`需要排除公司节假日和特殊原因引起的不能服务的情况，可以在后台管理中指定不可接受预约的时间范围，和原因，如果此时已经有预约的情况，需要通过用户留下来的电话等联系方式通知对方改期或取消预约，并从后台管理中将此订单做出相应的处理`
> 
> 2. 当天下午4点以后只能安排第二天的预约。
> 
> 3. 未来根据业务扩展的情况可以考虑员工产能进行排产，第一阶段可以暂时不用考虑这个问题。
> 
> 4. 排班需要考虑安装工数量，设备情况，服务平均执行时间，来安排计划。
>
> 5. 如果通过后台来调整预约时间，导致的预约订单修改，需要通过短信或者站内信通知用户
>
> 6. 预约的商品服务需要是库存余额中有货的产品，如果库存余额小于等于0，应该表示为无货（`应该拒绝下单还是，允许下单但通过人工确认后取消订单还是需要与客户沟通`）


4. 生成一张预约订单，并提供一个随机的订单号，保存在用户当前订单中，订单是待审核状态，用户可以选择取消订单，取消后订单状态为已取消。
5. 管理员在控制台可以查到已经处于待审核状态的订单，并根据实际工作情况进行确认，主要是考虑产能和库存情况，如果能够正常履约，将订单置为已审核状态，通过短信或者站内信推送给客户，并在前一天系统提醒管理员与用户进行确认。如果不能正常履约，需要提供相应的拒绝原因，并通过电话或者短信跟用户确认，得到确认后可以将订单置为已取消。
6. 至此预约完成。

**其他需求**

预约功能如果做好的话对用户的体验提升比较明显，整个流程的易用性需要比较好，过程流畅，尽量全面的覆盖各种业务场景，这里也需要跟客户做比较全面的沟通

### 用例UC06 履约

**参与者，及其关注点**

1. 管理员：系统推送预约订单到管理员控制台，管理员安排工单计划，**提前通知库存准备需要的轮毂和配件**，并在客户到店后，付费并安排安装工提供安装和服务，安装完成后将预约订单状态置为完成，客户可以为此次服务做评价，如果是现场评价，应该好评的几率比较大
2. 注册用户：到店接受改装和保养服务，向工作人员出示订单号或者手机号，查询订单并进行付费和安装。订单完成后，可以为此次服务进行评价，可选提出评价，这些评价可以做为改进服务的依据

**前置条件**

1. 已经建立预约订单，并且订单已审核
2. 客户到店接受服务

**完成依据**

改装或保养服务执行完成，预约订单处于完成状态。用户提出评价。

**主要场景**

1. 在预约单约定好的日期前一天，系统提醒第二天预约的预约订单情况，提前准备商品，如果有无法履约的情况，可以提前通知客户延期，详见[UC05 预约](#UC05) 中改期的部分，如果能够履约，提前通知仓库准备轮毂和配件
2. 客户驾车到店接受服务，向店内员工出示预约单号，或者手机号
2. 店员根据根据用户提供的信息查到预约单，收费，并安排工作人员进行安装
3. 安装完成后，店员将预约订单置为完成状态
4. 客户可能现场为此次服务进行评价，或者提出改进意见
5. 订单完成后超过30天关闭评价
